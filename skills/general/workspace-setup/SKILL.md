# Workspace setup (mise + uv)

## Required inputs

- Repo root path (default: current working directory)
- Target tool versions (defaults: Node `24`, Python `3.14`, Bun `latest`, uv `latest`)
- Whether to **apply** changes to the repo root or generate to `temp/` for review first
- OS/shell constraints (macOS/Linux; Windows requires WSL2)
- Whether internet access is allowed (mise typically download tool artifacts)

## Workflow

1) Ensure workspace-root agent guidance is present and up to date:

   - Confirm the repo has an `AGENTS.md` at the workspace root.
   - If it needs to be created or polished, use `references/agent-rules.md` as the authoring guide.

2) Ensure key workspace folders and local rules exist:

   - Confirm `temp/` exists and contains `temp/AGENTS.md`.
   - Confirm `artifacts/` exists and contains `artifacts/AGENTS.md`.
   - Ensure these per-folder `AGENTS.md` files are tracked in git even if the rest of the folder is ignored.

3) Generate config files (safe default: write to `temp/`):

   - From the skill folder:
     - `cd skills/general/workspace-setup`
     - `bash scripts/init_workspace.sh`
   - Alternative (from repo root): `bash skills/general/workspace-setup/scripts/init_workspace.sh`
   - This also generates a `.ignore` file that *re-includes* everything excluded by `.gitignore` so Codex can recognize all workspace files (per openai/codex#2952 guidance).
   - Review what was generated in `temp/workspace-setup/` before applying.

4) Apply to the repo root when ready:

   - From the skill folder:
     - `cd skills/general/workspace-setup`
     - `bash scripts/init_workspace.sh --apply`
   - Alternative (from repo root): `bash skills/general/workspace-setup/scripts/init_workspace.sh --apply`
   - With `--apply`, the script never overwrites existing files (`.mise.toml`, `.gitignore`, `.ignore`, `package.json`, `pyproject.toml`).
   - If any of those files already exist, generate to `temp/workspace-setup/` first, then:
     - **Backup** the current file (keep it under `temp/workspace-setup/backups/`).
     - **Merge** the generated file into the existing one, **always respecting existing values** (treat generated as defaults; only add missing keys/sections/lines).
     - Replace the repo-root file with the merged result.
   - Always review GitHub’s official `.gitignore` templates for your real stack and add missing patterns as needed: https://github.com/github/gitignore

## Assets

- Template files live in `assets/templates/` and are rendered by `scripts/init_workspace.sh`:
  - `.mise.toml.tmpl`
  - `package.json.tmpl`
  - `pyproject.toml.tmpl`
  - `gitignore.recommended`

5) Install toolchain with mise:

   - Ensure `mise` is installed and activated (shell hook) per https://mise.jdx.dev/getting-started.html
   - Run `mise install`

6) Initialize Node workspace (non-interactive `npm init` equivalent):

   - Ensure `package.json` exists (generated by the script); then use `npm` as usual

7) Create a project virtualenv and sync Python deps with uv:

   - Run `uv venv --seed`
   - Run `uv sync`
   - For one-off installs, prefer `uv pip install <pkg>` (or activate `.venv` before using `pip install ...`)

## Automation vs manual edits

- Prefer `scripts/init_workspace.sh` when you want a fast, deterministic baseline that is easy to reproduce across dev machines and CI.
- Prefer the plain-text workflow (and manual merges) when the repo already has workspace files or needs non-standard structure (monorepo, existing toolchain, custom ignores).

## Troubleshooting

- If any setup step fails: stop, clean the workspace, and report the failure in plain language with exact commands/log output and next-step suggestions.

## Outputs

- `.mise.toml` (tool versions + helper tasks)
- `.gitignore` (workspace ignore patterns; generated candidate if one already exists)
- `.ignore` (Codex file-visibility overrides; re-includes paths ignored by `.gitignore`)
- `package.json` (Node workspace manifest)
- `pyproject.toml` (Python deps; uv reads/writes this)
- `temp/workspace-setup/` (generated previews when not using `--apply`)
  - If you created or updated `AGENTS.md`, it should be at the workspace root (not under `temp/`).
- `temp/AGENTS.md` (rules for scratch outputs)
- `artifacts/AGENTS.md` (rules for deliverables)

## Definition of done

- `mise --version` works and `mise install` succeeds
- `node -v` reports `v24.x`
- `bun --version` prints a version (any)
- `package.json` exists in the repo root
- `.gitignore` exists in the repo root (or you merged the generated candidate)
- `.ignore` exists in the repo root
- `python --version` reports `3.14.x` (or the closest available version you intentionally chose)
- `uv --version` works and `uv sync` succeeds
- `.venv/` exists in the repo root after `uv venv --seed`
- `temp/` exists and contains `temp/AGENTS.md`
- `artifacts/` exists and contains `artifacts/AGENTS.md`

## Safety / quality checklist

- Do not overwrite existing `.mise.toml` or `pyproject.toml` without reviewing diffs.
- Do not add secrets to `pyproject.toml` or `.mise.toml`.
- `.ignore` may make locally-ignored files (including `workers.jsonc`) visible to the agent; do not read/print secrets unless explicitly required.
- Keep generated outputs in `temp/` unless intentionally applying to the repo.
